# Upservice MCP Server

*[Read this in English](README.md)*

MCP-сервер (Model Context Protocol), который оборачивает [публичный API Upservice](https://public.upservice.io/redoc), чтобы Claude (или любой другой MCP-клиент) мог напрямую управлять сотрудниками, проектами, спринтами, задачами, тегами, справочниками и сообщениями в каналах.

Собран на основе OpenAPI-спека `https://public.upservice.io/openapi.json`, сверен с живым API.

## Что умеет

58 инструментов, покрывающих:

- **Сотрудники** — список сотрудников
- **Проекты** — список, создание, получение, обновление, удаление, назначение менеджеров/участников, завершение
- **Спринты** — список, создание, получение, обновление, удаление, завершение, активация, добавление задач
- **Теги** — список, создание, обновление, удаление, привязка/отвязка к задачам/чатам/активам/контактам/вложениям
- **Задачи** — список/поиск, создание, получение, обновление, удаление, вложения, смена статуса, оценка трудозатрат, worklog, действия по согласованию (approval), со-исполнители, листы согласования/ознакомления
- **Справочники** (кастомные каталоги) — список/создание/получение/обновление/удаление справочников и их записей, плюс массовое управление связями (привязка записей к задачам, проектам, заказам и т.д.)
- **Каналы и файлы** — загрузка/отправка сообщений в чат, отправка сообщений в канал, загрузка файлов, получение ссылок на скачивание файлов

Каждый пишущий инструмент явно описывает документированные поля (с валидацией), и большинство также принимают словарь `extra_fields` для продвинутых/редких полей, которые задокументированы в API Upservice, но не смоделированы по отдельности.

**Упоминание сотрудников:** Upservice превращает `@Имя` в настоящий, уведомляющий меншен только если это записано как `@[Имя](employee_id)`. Инструменты, которые пишут текст (`upservice_send_chat_message`, `upservice_send_channel_message`, `upservice_create_task`, `upservice_update_task`), принимают поле `mentions: [{employee_id, display_name}]` — поставьте в тексте плейсхолдер `{{employee_id}}`, и сервер сам подставит нужный синтаксис.

**Известное ограничение API:** `GET /v1/tasks` (используется в `upservice_list_tasks`) не поддерживает фильтр `status`/`is_completed` — это подтверждено на живом API, а не только по документации. Фильтруйте по `status` на клиенте, предварительно сузив выборку через `date_end_gte`/`date_end_lte`, `project`, `author` или `responsible`.

## Установка

Сначала получите API-ключ Upservice: в аккаунте Upservice → настройки аккаунта → раздел API. Ключи выглядят так: `UPS-XXXX-XXXX-XXXX-XXXX`.

### Вариант A — `uvx` (рекомендуется, без ручной установки)

Требуется установленный [`uv`](https://docs.astral.sh/uv/) (`brew install uv` или см. их документацию). Добавьте в конфиг вашего MCP-клиента (например, `claude_desktop_config.json`):

```json
{
  "mcpServers": {
    "upservice": {
      "command": "uvx",
      "args": ["--from", "/absolute/path/to/upservice_mcp", "upservice-mcp"],
      "env": {
        "UPSERVICE_API_KEY": "UPS-XXXX-XXXX-XXXX-XXXX"
      }
    }
  }
}
```

Если проект опубликован в git-репозитории, `--from` может указывать прямо на него вместо локального пути:
`"args": ["--from", "git+https://github.com/alexherbaly/upservice-mcp", "upservice-mcp"]` — тогда коллеге нужен только URL и свой собственный API-ключ, без копирования файлов.

Лучше закрепиться на ref, а не на голом URL — без него `uvx` следует за веткой `main`, и неудачный пуш в `main` сломает сервер у всех коллег при следующем запуске. Два варианта:

- **`@stable`** (рекомендуется для коллег) — тег, который CI сам двигает на верхушку `main` при каждой успешной сборке. Не нужно помнить про релиз ради рядовых фиксов — у всех сервер сам подхватит последний рабочий коммит при следующем перезапуске.
- **`@v0.2.0`** (конкретная версия) — навсегда зафиксирована на этом коммите, для случаев, когда нужно именованное состояние, на которое можно сослаться позже (release notes, «версия, которую тестировали такого-то числа»). Такие теги ставьте вручную через `git tag`, когда это оправдано, а не на каждый пуш.

Пример: `"args": ["--from", "git+https://github.com/alexherbaly/upservice-mcp@stable", "upservice-mcp"]`

`uv` сам собирает и кеширует изолированное окружение при первом запуске; ничего не ставится в систему.

### Вариант B — обычный venv + pip

```bash
cd upservice_mcp
python3 -m venv venv
venv/bin/pip install -r requirements.txt
```

```json
{
  "mcpServers": {
    "upservice": {
      "command": "/absolute/path/to/upservice_mcp/venv/bin/python3",
      "args": ["/absolute/path/to/upservice_mcp/src/upservice_mcp/server.py"],
      "env": {
        "UPSERVICE_API_KEY": "UPS-XXXX-XXXX-XXXX-XXXX"
      }
    }
  }
}
```

После этого перезапустите Claude Desktop. Инструменты появятся с префиксом `upservice_`.

### Опционально: кастомный base URL

Если Upservice когда-нибудь сменит хост API, или вы используете приватный/on-prem инстанс, переопределите его:

```json
"env": {
  "UPSERVICE_API_KEY": "UPS-XXXX-XXXX-XXXX-XXXX",
  "UPSERVICE_API_BASE_URL": "https://public.upservice.io"
}
```

## Локальное тестирование

```bash
# uvx
UPSERVICE_API_KEY=UPS-XXXX uvx --from . upservice-mcp

# venv
UPSERVICE_API_KEY=UPS-XXXX venv/bin/python3 src/upservice_mcp/server.py
```

Или интерактивно через MCP Inspector:

```bash
npx @modelcontextprotocol/inspector uvx --from . upservice-mcp
```

## Заметки

- Авторизация использует «сырой» API-ключ в заголовке `Authorization` (не `Bearer`-токен) — сервер делает это за вас.
- Разрушительные инструменты (`delete_*`, `unassign_tag`) помечены `destructiveHint: true`, чтобы MCP-клиенты могли предупреждать пользователя.
- У некоторых тел запросов, задокументированных Upservice, много опциональных полей; там, где инструмент не моделирует поле явно, передавайте его через `extra_fields` (обычный JSON-объект) — оно будет добавлено в тело запроса.
- **Никогда не делитесь своим `UPSERVICE_API_KEY`** — каждый должен сгенерировать собственный ключ в настройках своего аккаунта Upservice.
